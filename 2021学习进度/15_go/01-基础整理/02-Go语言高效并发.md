Go语言高效并发

### 1.并发的基础：

Goroutines和Channels的申明和使用

#### 1.1相关概念

- 进程：在操作系统中，进程是一个非常重要的概念。当你启动一个软件/程序的时候，操作系统会为这个软件创建一个进程，这个进程就是这个程序的工作空间，它包含了软件运行的时候所需的所有的资源，比如内存空间，文件句柄等
- 线程：线程是进程的执行空间，一个进程可以有多个线程，线程被操作系统调度执行，比如下载一个文件，发送一个消息等。这种多个线程被操作系统同时调度执行的情况，就是多线程的并发
- 协程：Go语言中没有线程的概念，只有协程，也称为goroutine。相对于线程来说，协程更加轻量，一个程序可以随意启动成千上万个goroutine。goroutine被Go runtime所调度。这一点和线程不一样，也就是说，go语音的并发是由go自己所调度的，自己决定自己同时执行多少个goroutine。

#### 1.2 启动一个goroutine

go关键字语法:go 关键字后面跟一个方法或者函数的调用，就可以启动一个goroutine。

```
//go function()
func main() {
	go fmt.Println("hufan say")
	fmt.Println("hello hufan , welcome to go word")
	time.Sleep(time.Second)
}
//上面这段代码中有两个goroutine。一个是main函数启动的main goroutine 另一个是我自己用go关键字启动的goroutine
//输出结果为:
hello hufan , welcome to go word
hufan say
```

#### 1.3协程的通信：Channel

申明一个channel：

chan是一个关键字，表示是channel类型。后面的string表示channel里的数据是string类型。

```
ch := make(chan string)
```

channel的使用：接受的时候，符号<- 在channel的 左边。发送的时候符号 <- 在channel的右边

**接收：**获取chan中的值，操作符好为<-chan 

**发送：**向chan发送值，把值放在chan中，操作符为chan <-

e.g

```
func main() {
	ch := make(chan string)
	go func() {
		fmt.Println("cr7 is the best football palyer in the world")
		ch <- "channel 发送完成"
	}()
	fmt.Println("我是main函数启动的goroutine")
	str := <-ch
	fmt.Println("接收到的channel中的消息为：",str)
}
```

##### 1.3.1 无缓冲channel

上面的例子中就是一个无缓冲的channel。它的容量是为0的，不能储存任何数据，意味着，数据并不会在channel中做任何停留，所以无缓冲channel只能起到数据传输的作用。这种无缓冲的channel的发送和接收操作是同时进行的，它也可以被称之为同步channel。

##### 1.3.2 有缓冲channel

有缓冲的channel类似于一个可阻塞的队列，内部的元素是安装先进先出的。

- 有缓冲 channel 的内部有一个缓冲队列；

- 发送操作是向队列的尾部插入元素，如果队列已满，则阻塞等待，直到另一个 goroutine 执行，接收操作释放队列的空间；

- 接收操作是从队列的头部获取元素并把它从队列中删除，如果队列为空，则阻塞等待，直到另一个 goroutine 执行，发送操作插入新的元素。

e.g

```
ch := make(chan int,5)
```

如上，我申明了一个channel，channel的数据类型为int，channel的容量为5。

下面我演示下基本的使用：

e.g

```
func main() {
	cacheCh := make(chan int,5)
	cacheCh <- 7
	cacheCh <- 9
	fmt.Printf("这个cache channel的容量为:%d 元素个数为：%d", cap(cacheCh), len(cacheCh))
}
//输出
这个cache channel的容量为:5 元素个数为：2

```

##### 1.3.3 单向channel

针对有些特殊业务场景，例如限制某个channel只能接收，不能发送。又或者只能发送不能接收。

```
onlySend := make(chan<- int)

onlyReceive:=make(<-chan int)

```

##### 1.3.4 关闭channel

如果channel关闭了，则不能再向里面发送数据了，如果发送的话，会引起painc异常。但是可以接受channel里的数据，如果channel里没有数据的话，那么接受到的值为元素类型的零值。

```
close(cacheCh)
```

这种channel的多路复用。

可以简单的这么理解：N个channel中，任意一个channel中有数据产生，那么select都可以监听到，然后执行相应分支，接收数据并处理数据。

e.g

### 2并发的安全

在go语言中除了channel这种比较容易而且高级的同步机制（内部使用了互斥锁来保证并发的安全）。还可以使用sync.Mutex,sync.WaitGroup等比较原始的同步机制。

#### 2.1资源竞争

在一个goroutine中，如果分派的内存没有被其他的goroutine访问，只在该goroutine中被使用，那么不存在资源竞争的问题。

但是如果同一块内存被多个goroutine同时访问，那么就会出现无法预料的问题。

e.g

```
var sum = 0
func main() {

	for i := 0; i < 1000; i++ {
		go add(10)
	}
	time.Sleep(2*time.Second)
	fmt.Println("和值为",sum)
}
func add(i int){
	sum += i
}
//预期的结果为10000，但是结果可能是9870或者其他的
```

问题分析：

因为sum是共享的资源。会有同时多个协程去操作sum+=i

解决办法：引入互斥锁 sync.Mutex。对sum+=i 进行枷锁保护。

e.g

```

var (
	sum int
	mutex sync.Mutex
)
func main() {
	startTime := time.Now()
	for i := 0; i < 100000; i++ {
		go add(10)
	}
	time.Sleep(2 * time.Second)
	fmt.Println("和值为",sum)
	endTime := time.Now()
	fmt.Println("执行100000次sum操作，耗时",endTime.Sub(startTime))
}
func add(i int){
	mutex.Lock()
	defer mutex.Unlock()
	sum += i
}
```

#### 2.2读写锁

sync.RWMutex:可以通过读写锁来降低锁的粒度，来优化程序的性能

e.g

我新增了一个readSum的函数。同时启动了十个协程去读取sum的值，但是由于readSum函数没有任何的加锁控制，所以它不是并发安全的，因为一个协程在执行add的时候，另外一个协程可能正在执行readSum。

参照上面的，可以用 sync.Mutex来解决

```
var (
	sum int
	mutex sync.Mutex
)
func main() {
	startTime := time.Now()
	for i := 0; i < 100000; i++ {
		go add(10)
	}
	for i := 0; i <10000 ; i++ {
		go fmt.Println("和值为",readSum())
	}
	time.Sleep(10 * time.Second)
	fmt.Println("和值为",sum)
	endTime := time.Now()
	fmt.Println("执行100000次sum操作，耗时",endTime.Sub(startTime))
}
func add(i int){
	mutex.Lock()
	defer mutex.Unlock()
	sum += i
}
func readSum()int{
	mutex.Lock()
	defer mutex.Unlock()
	return sum
}
```

这样虽然能解决问题。但是性能好低呀。因为readSum和add是使用的同一个sync.Mutex。所以他们的操作是互斥的。也就是一个 goroutine 进行修改操作 sum+=i 的时候，另一个 gouroutine 读取 sum 的操作 b:=sum 会等待，直到修改操作执行完毕。

性能不好，怎么办咧。优化撒 。

先分析场景：

- 写的时候不能同时读
- 读的时候不能同时写
- 读的时候可以同时读

分析清楚了，那就上解决方案。读写锁sync.RWMutex

```
var (
	sum int
	mutex sync.Mutex
	rwMutex sync.RWMutex
)
func main() {
	startTime := time.Now()
	for i := 0; i < 100000; i++ {
		go add(10)
	}
	for i := 0; i <100000 ; i++ {
		go fmt.Println("和值为",readSum())
	}
	time.Sleep(10 * time.Second)
	fmt.Println("和值为",sum)
	endTime := time.Now()
	fmt.Println("执行100000次sum操作，耗时",endTime.Sub(startTime))
}
func add(i int){
	mutex.Lock()
	defer mutex.Unlock()
	sum += i
}
func readSum()int{
	//我只获取读锁
	rwMutex.RLock()
	defer rwMutex.RLock()
	return sum
}
```

#### 2.3 sync.WaitGroup

上面我写的测试代码中，有这么一句话time.Sleep(10 * time.Second) .

这块之所以这么写是为了防止主函数main返回使用。一旦main函数返回了，程序也就退出了。

但是我们不知道上面程序运行的协程什么时候能完全执行成功，所以设置了一个比较长的等待时间。

但是这里就又引入了一个新的问题：

如果程序的执行时间超过10s，但是设置的等待时间只有10s，程序会提前返回，会导致不可预知的后果;

如果程序执行的时间不超多10s，超长的等待时间，又造成性能的问题。

此时我们就呼唤一个英雄了撒。需要一个方法能监听所有协程的执行，一旦全部执行，程序马上退出，这样既可以让所有协程全部执行完毕，也可以及时退出，节省时间，提升性能。

解决办法：

- channel
- sync.WaitGroup（强烈推荐用这个）

e.g

使用方法：

```
1.声明一个 sync.WaitGroup，然后通过 Add 方法设置计数器的值，需要跟踪多少个协程就设置多少；

2.在每个协程执行完毕时调用 Done 方法，让计数器减 1，告诉 sync.WaitGroup 该协程已经执行完毕；

3.最后调用 Wait 方法一直等待，直到计数器值为 0，也就是所有跟踪的协程都执行完毕。
```

```
func main()  {
	run()
}
func run()  {
	var wg sync.WaitGroup
	wg.Add(200000)
	for i := 0; i < 100000; i++ {
		go func() {
			defer wg.Done()
			Add(10)
		}()
	}

	for i := 0; i < 100000; i++ {
		go func() {
			defer wg.Done()
			fmt.Println("和为：",readSum())
		}()
	}
	wg.Wait()

}
```

#### 2.4 sync.Once

常常会有这个需求：

让代码只执行一次，哪怕是在高并发的情况下，比如创建一个单例。针对这种情形，Go 语言为我们提供了 sync.Once 来保证代码只执行一次。

```
func DoOnce(){
	var once sync.Once
	onceBody := func() {
		fmt.Println("this body on run one times")
	}
	//用于等待协程执行完毕
	done := make(chan bool)

	for i := 0; i < 100; i++ {
		go func() {
			once.Do(onceBody)
			done <- true
		}()
	}

	for i := 0; i < 100; i++ {
		<-done
	}
}
```

#### 2.5 syncCond

从字面意思理解是条件变量，它具有阻塞协程和唤醒协程的功能。

- Wait 阻塞当前协程，一直到被其他协程调用Broadcast或者Signal方法唤醒。使用的时候需要加锁，使用sync.Cond中的锁就可以了。
- Signal:唤醒一个等待时间最长的协程
- Bradcast:唤醒所有的协程

备注：在调用Signal或者Broadcast前需要确保目标协程处于Wait阻塞状态。不然会出现死锁的问题。

#### 3.Context

先简单介绍下Context。这个是Go语言库为我们提供的跟踪协程的方案。

需求场景：例如，我想要退出一个协程，应该怎么完成呢？

```
func WatchDog(stopCh chan bool,name string){
	//开启for select循环，一直在后台监控
	for {
		select {
		case <-stopCh:
			fmt.Println(name,"收到停止命令，马上停止")
			return
		default:
			fmt.Println("watch dog is working")
		}
		time.Sleep(1*time.Second)
	}
}

func main()  {
	var wg sync.WaitGroup
	stopCh := make(chan bool)
	wg.Add(1)
	go func() {
		defer wg.Done()
		test.WatchDog(stopCh,"监控狗001")
	}()
	time.Sleep(5*time.Second)
	stopCh <- true
	wg.Wait()
}
```

这种select + channel的方式可以优雅的停止协程，但是同样也有局限性。例如我要同时取消多个协程，又或者我需要定时取消协程。

所以Context就可以帮我们解决这个问题。

e.g

​	context.WithCancel(context.backgroud()) 可以生成一个可以取消的context，用于发送停止命令。

​	context.backgroud() 用于生成一个空的Context，一般作为这个Context树的根节点。

```
func main()  {
	var wg sync.WaitGroup
	wg.Add(1)
	ctx,stop := context.WithCancel(context.Background())
	go func() {
		defer wg.Done()
		test.WatchDog(ctx,"监控狗001")
	}()
	time.Sleep(5*time.Second)
	stop()//发出停止命令
	wg.Wait()
}

func WatchDog(ctx context.Context,name string){
	//开启for select循环，一直在后台监控
	for {
		select {
		case <-ctx.Done():
			fmt.Println(name,"收到停止命令，马上停止")
			return
		default:
			fmt.Println("watch dog is working")
		}
		time.Sleep(1*time.Second)
	}
}
```

再这里在仔细介绍下我们的主角Context

Context是一个接口，它具备手动、定时、超时、发出取消信号、传值等功能。主要用于控制多个协程之间的协作，尤其是取消操作。

一旦取消指令下达之后，那么被context跟踪的这些协程都会受到取消信号，就可以做清理和退出操作。

```
type Context interface {
	Deadline() (deadline time.Time, ok bool)
	Done() <-chan struct{}
	Err() error
	Value(key interface{}) interface{}
}
```

Deadline:可以用来获取设置的截止时间，第一个返回值deadline是截止时间，到了这个时间点，Context会自动发起取消请求，第二个ok代表是否设置了截止时间。

Done:方法返回了一个只读的channel，类型为struct{}。在协程中，如果该方法返回的chan可以读取，则意味Context已经发起了取消信号。通过Done方法收到这个信号后，就可以做清理操作，然后退出协程，释放资源

Err:方法返回取消的错误原因，也就是什么原因Context被取消。

Value：方法获取该Context上绑定的值，是一个键值对。所以要通过一个key才可以获取对应的值。

##### 3.1Context树

go语言为我们提供了函数帮我们生成不同的Context。通过这些函数，我们可以生成一颗Context树，这样Context才可以关联起来。

从使用功能上来说，有四种实现好的Context的

- 空Context：不可取消，没有设置时间，主要用于Context树的根节点
- 可取消的Context：用于发出取消信号，当取消的时候，它的子Context也会取消
- 可定时取消的Context：多了一个定时的功能
- 值Context：用于储存的key-value键值对

e.g

- WithCancel(parent Context):用来生成一个可取消的Context
- WithDeadline(parent Context,d time.Time):生成一个可定时取消的Context，参数d 用来取消的具体时间
- WithTimeout(parent Context,timeout time.Duration)：生成一个可以超时取消的Context，参数timeout用于设置多久之后取消
- WithValue(parent Context,key,val interface{})：生成一个可携带key-value键值对的Context



下面我演示下使用Context进行传值

e.g

```
func main()  {
	var wg sync.WaitGroup
	wg.Add(4)
	ctx,stop := context.WithCancel(context.Background())
	varContext :=context.WithValue(ctx,"userUuid",007)
	go func() {
		defer wg.Done()
		getUser(varContext)
	}()
	go func() {
		defer wg.Done()
		test.WatchDog(ctx,"监控狗001")
	}()
	go func() {
		defer wg.Done()
		test.WatchDog(ctx,"监控狗002")
	}()
	go func() {
		defer wg.Done()
		test.WatchDog(ctx,"监控狗003")
	}()

	time.Sleep(5*time.Second)
	stop()//发出停止命令
	wg.Wait()
}

func getUser(ctx context.Context){
	for  {
		select {
		case <- ctx.Done():
			fmt.Println("获取用户：","协程退出")
			return
		default:
			userUuid := ctx.Value("userUuid")
			fmt.Println("获取用户","用户uuid为：",userUuid)
			time.Sleep(1*time.Second)
		}
	}
}
```

##### 3.2 Context的使用原则

- Context不要放在结构体中，要以参数的形式传递
- Context作为函数的参数的时候，要放在第一个位置
- 要使用Context.Backgroup函数生成的根节点的Context，也就是要使用顶层的Context
- Context传值要传递必须的值，而且尽可能的少些
- **Context多协程安全，可以在多个协程中放心使用**
