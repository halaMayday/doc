#### 1.运行时指令重排

**指令重排的场景**：当cpu写缓存的时候 发现缓存区块正在被其他cpu使用，为了提高cpu的性能，可能将后面的读缓存命令优先执行。

eg：

```
//代码
x = 100;
y = z;
//正常执行的三步骤
1：将100写入x
2：读取z的值
3：将z的值写入y
//重排后的执行顺序
1：读取z的值
2：将z的值写入y
3：将100写入x
```



并非随便重排，需要遵守as-if-serial语义

意思是：编译器和处理器为了提高并行度（不管怎么重排序），（单线程）程序的执行结果不能被改变。编译器，runtime和处理器都必须遵守as-if-serial语义。

换句话说：就是编译器和处理器不会对存在依赖关系的操作做重排序。



#### 2.两个问题

##### 2.1 CPU高速缓存

缓存中的数据与主内存的数据并不是实时同步的，各个CPU间缓存的数据也是实时同步。在同一时间点，各CPU看到的同一内存地址的数据的值可能是不一致的。

2.2 指令重排的局限性

​	虽然遵守了as-if-serial语义，单仅在cpu自己执行的情况下能保证结果正确。在多核多线程下，指令逻辑无法分辨因果关联，可能出现乱序执行，导致程序执行结果错误。

#### 3. 解决办法

##### 3.1 内存屏障

写内存屏障（Store memory Barrier）：在指令后插入Store Barrier，能让写入缓存中的最新数据写入主内存，其他线程可见。

读内存屏障（Load Memory Barrier）: 让cpu缓存和主内存保持一致，避免了缓存导致的一致性问题。
